# third 能力更改方式

本文给 Codex 使用。遇到 third 能力变更需求时，先判断改 `Prompt`、`Workflow Template` 还是 `Tool`。

## 先判断改哪里

| 需求 | 改哪里 |
|---|---|
| 只是模型理解不准、字段选择不稳、输出措辞不好 | 改 `Prompt` |
| 要新增或调整 workflow 步骤、顺序、artifact 依赖、确认门 | 改 `Workflow Template` |
| 要调用新的外部能力，或改变真实执行逻辑 | 改 `Tool` |
| 要新增 `prompt_ref`，但执行逻辑与现有 Agent 不同 | 改 `Prompt` + `agent_runner.py` |
| 写入、删除、字段变更的安全规则要变 | 改 `validation.py`，必要时改 Tool 和 Template |

## 改 Prompt

业务 Agent prompt：

1. 改 `third/Prompt/runagent/*.yaml`。
2. 保持 `prompt_key`、`agent_name`、`input_schema_json`、`output_schema_json` 与运行时契约一致。
3. 如果新增 YAML，确认 `workflow/agent_runner.py` 支持该 `prompt_ref`。
4. 本地直接 Python 运行时执行 seed；生产私有 Compose 由 `third-prompt-seed` 在启动链路中自动执行：

```bash
python -m third.scripts.seed_runagent_prompts
```

5. 更新 `third/Prompt/runagent/README.md`。
6. 补测试：seed、Agent Runner 输出结构、相关 workflow。

workflowagent prompt：

1. 改 `third/Prompt/workflowagent.yaml`。
2. 只用于影响 LLM 模式的模板选择或旧版完整 plan 输出。
3. 如果规则模式也要识别，改 `third/agents/workflowagent/agent.py`。
4. 补测试：规则模式和 LLM 输出兼容解析。

不要只改 Prompt 的情况：

- 新增 Tool。
- 改 `workflow_plan.steps` 结构。
- 改 artifact key。
- 改 validation 或 Tool 需要的 JSON 结构。

## 改 Workflow Template

模板入口是 `third/workflow/registry/templates.py`。

执行步骤：

1. 在 `TEMPLATE_CATALOG` 增加或修改 `template_key`、`intent`、`risk_level`、`purpose`、`step_order`。
2. 在 `build_plan_from_template()` 增加或修改分支。
3. 写 builder，生成完整 `workflow_plan`。
4. 每个 step 必须明确：
   - `step_id`
   - `kind`
   - `tool_name` 或 `agent_name/prompt_ref`
   - `input.from_session`
   - `output.save_as`
   - 必要的 `validation`
5. 如果是写入、副作用、删除、字段变更，必须包含 validation 和 confirm。
6. 如果新增 artifact 依赖，确认前序 step 已产生该 artifact。
7. 更新 `third/workflow/plan_validator.py`，拒绝缺步骤、错顺序、错 artifact 的 plan。
8. 更新 `third/Prompt/workflowagent.yaml`，告诉 LLM 什么时候选择该 `template_key`。
9. 必要时更新 `third/agents/workflowagent/agent.py` 的规则模板识别。
10. 补测试：模板能生成合法 plan；删除关键步骤会被 validator 拒绝。
11. 更新 `third/codex.md`、`third/README.md` 和相关 `third/docs/`。

不要让 LLM 自由拼复杂长流程。优先让 workflowagent 输出 `template_key`，由 builder 生成 steps。

## 改 Tool

Tool registry 不是实现。只改 `workflow/registry/tools.py` 不会新增可执行能力。

执行步骤：

1. 在 `third/Tool/` 新增或修改 Tool 实现。
2. 在 `third/workflow/tool_dispatcher.py` 注册 `tool_name`。
3. 在 `third/workflow/registry/tools.py` 更新 Tool 能力目录。
4. 如果 Tool 会写入、删除、改字段或影响外部系统，更新 `third/workflow/validation.py`：
   - 校验输入结构
   - 校验字段和类型
   - 生成确认预览
   - 生成幂等 key
   - 拒绝危险或不完整请求
5. 如果 Tool 需要新流程，更新 Workflow Template。
6. 如果 Tool 需要外部客户端能力，更新对应 client 或 mock repository。
7. 补测试：dispatcher 注册、registry 一致、validation、mock Tool 流程、失败场景。
8. 更新 `third/README.md`、`third/codex.md`、`third/docs/`。

## mistakeagent

第一版只做只读复盘建议器。

它应该输出：

- 应该改 Prompt、Template、Tool、Validator、测试还是文档。
- 需要新增或修改的 `prompt_key`、`template_key`、`tool_name`。
- 建议的测试场景。

它不应该：

- 运行时直接改 Python 文件。
- 直接改 registry。
- 绕过 validation、confirm、测试。
- 自动新增可执行 Tool。

## 提交前检查

文档改动：

```bash
rg "旧文件名|旧模板名" third
git diff -- third
```

Prompt 改动：

```bash
python -m third.scripts.seed_runagent_prompts
python -m unittest third.tests.test_runagent_registry
```

Template 或 Tool 改动：

```bash
python -m unittest discover third\tests
```
