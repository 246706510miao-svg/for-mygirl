# third Agent-Led 外部调用契约

本文档给 Codex 和外部业务后端使用。正式前端链路必须和 debug demo 使用同一套 third workflow 状态机：外部系统提交用户原文，third 决定 workflow，后端读取 third snapshot，后端选择展示字段，用户确认后 resume third。

## 状态机

```text
invoke: POST /workflows/invoke
status/snapshot: GET /workflows/{session_id} + GET /internal/workflows/{session_id}/snapshot
waiting_user: 后端返回 confirmation 给前端
resume: POST /workflows/{session_id}/resume
final/snapshot: GET /workflows/{session_id} + GET /internal/workflows/{session_id}/snapshot
```

`/debug` 只是托管了同一套状态机的调试页面，不是另一条业务流程。

## invoke 输入

```json
{
  "content": [
    {
      "text": "今天学习了韩语，写到飞书里面去"
    }
  ],
  "metadata": {
    "businessSessionId": "session_xxx",
    "idempotencyKey": "cmid_xxx"
  }
}
```

字段规则：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `content[0].text` | 是 | 用户原始输入。业务后端不要添加“生成草稿：”等固定前缀。 |
| `metadata.businessSessionId` | 是 | 外部业务会话 ID，例如 SpringBoot `RECORD_SESSION.id`。 |
| `metadata.idempotencyKey` | 是 | 外部幂等 ID，消息阶段使用 `clientMessageId`。 |
| `metadata.businessRecordId` | 可选 | 已有正式记录 ID，重试或追踪时使用。 |
| `metadata.operation` | 可选 | 只用于显式 override；默认不传，由 workflowagent 决定。 |

## 轻量状态接口

`POST /workflows/invoke` 和 `GET /workflows/{session_id}` 返回同一结构：

```json
{
  "session_id": "sess_xxx",
  "status": "queued | running | waiting_user | success | failed | cancelled",
  "confirmation": {
    "confirmation_id": "confirm_xxx",
    "request_text": "确认执行以下飞书写入操作吗？",
    "preview_json": {}
  },
  "content": [
    {
      "text": "最终文本答案"
    }
  ],
  "error_text": null
}
```

后端只用这个接口推进状态；需要字段对齐时必须读取 snapshot。

## outputs 字段

`outputs.draft` 来自 `artifact_key=record.draft`：

```json
{
  "title": "今日自律记录",
  "recordDate": "2026-06-22",
  "summary": "今天学习了韩语",
  "score": 80,
  "tags": ["学习"],
  "suggestion": "明天继续保持。",
  "previewText": "今天学习了韩语",
  "source": "llm"
}
```

`outputs.writePayload` 来自 `artifact_key=validation.write_payload`：

```json
  "operation": "create_record",
  "tool_name": "tool_CreateFeishuBitableRecord",
  "request_key": "create_request",
  "tool_input_payload": {
    "original_input": "今天学习了韩语，写到飞书里面去",
    "create_request": {
      "records": [
        {
          "fields": {
            "事项名称": "韩语学习",
            "总结": "今天学习了韩语"
          }
        }
      ]
    }
  },
  "idempotency_key": "create_record:feishu_bitable:hash",
  "payload_hash": "hash",
  "warnings": [],
  "preview": {
    "record_count": 1,
    "records": [
      {
        "record_id": null,
        "fields": {
          "事项名称": "韩语学习",
          "总结": "今天学习了韩语"
        },
        "lookup": {}
      }
    ]
  },
  "expires_at": "2026-06-29T10:00:05"
}
```

`outputs.writeResult` 来自 `artifact_key=write_result`，内容由具体 Tool 决定，后端必须整体保存到同步追踪数据中。

`outputs.tableSchema` 来自 `feishu.table_schema_after` 或 `feishu.table_schema`，用于后端排查字段对齐问题。

`outputs.records` 来自 `feishu.records` 或 `feishu.candidate_records`，用于查询、更新、删除类 workflow 的候选记录展示或追踪。

`outputs.finalAnswer` 来自 `artifact_key=final.answer`。

## 后端消费规则

- 后端必须保存 `businessSessionId <-> third session_id` 映射。
- 后端必须读取 snapshot，而不是直接猜 artifact key。
- 后端可以读取 `outputs.writePayload.preview` 生成用户确认预览。
- 后端可以读取 `outputs.draft` 或 `outputs.writePayload.preview` 生成本地 `RECORD_DRAFT`。
- 后端给前端的字段必须是后端选择后的稳定字段；third 原始 JSON 只保存到 trace/sync payload，不直接透给前端。
- `status=waiting_user` 时，后端必须把 `thirdSessionId` 和 `confirmationId` 返回给前端。
- 用户确认时，后端必须调用 `/workflows/{session_id}/resume`，不得绕过 third 确认门。

## resume 输入

```json
{
  "confirmation_id": "confirm_xxx",
  "approved": true,
  "content": [
    {
      "text": "业务前端已确认执行"
    }
  ]
}
```

`approved=false` 时 third 状态变为 `cancelled`，不会执行确认门后面的写入 Tool。

## override

默认不传 `metadata.operation`。当前只保留一个 override：

| `metadata.operation` | 行为 |
| --- | --- |
| `draft_generate` | 强制使用 `record_draft` template，输出 `outputs.draft`，不写飞书。 |
