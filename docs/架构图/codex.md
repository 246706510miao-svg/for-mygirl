# 架构图模块 Codex 入口

本文件是 `docs/架构图` 的主入口。Codex 需要判断系统边界、模块归属、部署方式或外部服务调用位置时，先读这里。

## 这个模块解决什么问题

- 系统由哪些端、哪些后端模块、哪些数据与外部服务组成。
- 新功能应该放在前端、后端、数据库、外部服务还是部署层。
- 哪些能力属于 MVP，哪些暂时不要做。
- 部署时公网、内网、配置、日志和数据库的边界在哪里。

## 一句话架构

这是一个个人使用的日常记录 Web 应用：用户在 iPhone Web/PWA 上通过文本或语音记录，管理员在 PC Web 后台配置每日内容和处理异常；前端只调用 Spring Boot 后端，后端统一编排 AI、ASR、飞书 MCP、本地数据库和外部配置。

## 固定技术选择

| 层级 | 技术或形态 | 说明 |
|---|---|---|
| 用户端 | TypeScript 移动 Web 或 PWA | 面向 iPhone Safari |
| 管理员端 | TypeScript PC Web | 可与用户端在同一个前端项目中用路由区分 |
| 后端 | Java Spring Boot | 系统控制中心 |
| 公网入口 | Nginx | HTTPS、静态文件、API 反向代理 |
| 数据库 | 单体业务数据库 | 不开放公网 |
| 外部能力 | 大模型、ASR、飞书 MCP | 只能由后端调用 |

## 架构边界

### 前端负责

- 用户端记录界面、文本输入、浏览器录音、草稿预览、确认写入、最近记录展示。
- 管理员端记录状态查看、每日内容配置、异常记录处理、用户端展示内容管理。
- 调用后端 API 并展示 `chatReply`、草稿、首页数据和管理后台数据。

### 前端禁止

- 直接调用大模型、语音识别服务或飞书 MCP。
- 保存 API Key、飞书 Token、模型密钥。
- 把飞书页面当成用户端预览来源。
- 在 MVP 中实现上传背景图、上传照片、长期保存语音音频等文件能力。

### 后端负责

- 身份识别、记录会话、AI 编排、语音处理、飞书 MCP 编排。
- 每日内容配置、最近记录展示、异常保存与重试、系统配置读取。
- 保存 RECORD_MESSAGE、RECORD_DRAFT、DAILY_RECORD、RECORD_DISPLAY、FEISHU_SYNC 等业务数据。
- 校验 AI 生成的飞书 MCP Payload，再决定是否调用飞书 MCP。

### 后端禁止

- 把可变配置或密钥硬编码到业务代码。
- 让 AI 直接决定飞书写入位置并绕过后端校验。
- 未校验就执行 AI 生成的 MCP Payload。
- 在 confirm 或 retry-sync 中无限循环调用 AI 修正 Payload。
- 暴露数据库端口到公网。

## 后端模块清单

| 模块 | 职责 | 主要关联 |
|---|---|---|
| 身份识别模块 | 区分用户和管理员 | 外部配置、APP_PERSON |
| 记录会话模块 | 管理一次对话式记录过程 | RECORD_SESSION、RECORD_MESSAGE、RECORD_DRAFT |
| AI 编排模块 | 生成草稿、摘要、标签、评分、MCP Payload | 大模型服务、Prompt、草稿上下文 |
| 语音处理模块 | 接收临时音频并调用 ASR | RECORD_MESSAGE.asr_text |
| 飞书 MCP 编排模块 | 调用飞书 MCP 写入目标文档或多维表格 | FEISHU_SYNC、APP_CONFIG |
| 每日内容配置模块 | 管理用户端每天展示的提示、主题、引导、卡片 | DAILY_CONTENT |
| 记录展示模块 | 为用户首页和最近记录生成本地展示数据 | RECORD_DISPLAY、DAILY_RECORD |
| 异常与重试模块 | 保存失败原因、payload 快照并支持重试 | FEISHU_SYNC、DAILY_RECORD |
| 系统配置模块 | 读取非敏感配置、Prompt、Schema | 外部配置、APP_CONFIG |

## 核心业务闭环

1. 用户在 iPhone 页面输入文本或录音。
2. 前端提交文本或临时音频给后端。
3. 后端对音频调用 ASR，得到 `asrText`。
4. 后端保存本轮 RECORD_MESSAGE。
5. 后端结合历史消息和当前草稿调用 AI。
6. AI 返回新的 RECORD_DRAFT，前端展示草稿预览和 `chatReply`。
7. 用户可以继续修改，直到确认某个草稿版本。
8. 后端使用 `clientConfirmId` 做确认幂等，锁定 session 和 draft。
9. 后端创建 DAILY_RECORD，并让 AI 根据飞书 Schema 生成 MCP Payload。
10. 后端校验目标、字段、类型、必填项和权限，最多修正有限次数。
11. 校验通过后调用飞书 MCP；成功或失败都保存 FEISHU_SYNC。
12. 后端生成或更新 RECORD_DISPLAY，用户端展示本地最近记录状态。

## 部署边界

MVP 使用一台云服务器即可。

| 部署单元 | 位置 | 约束 |
|---|---|---|
| Nginx | 公网入口 | 接收 HTTPS，请求静态文件或反向代理 API |
| 前端静态文件 | 云服务器 | 由 Nginx 托管 |
| Spring Boot | 云服务器内部端口 | 不直接暴露到公网 |
| 数据库 | 本机或内网 | 不开放公网端口 |
| 外部配置文件/环境变量 | 服务器 | 保存密钥、目标 ID、Token、模型配置 |
| 日志文件 | 服务器 | 保存错误、同步记录、异常排查信息 |
| 外部服务 | 第三方 | 仅由后端访问 |

## 暂时不做

- 原生 iOS App、小程序。
- 复杂多用户系统、复杂权限系统。
- 管理员统计图表。
- 飞书内容直接预览。
- 上传背景图、上传照片、服务器文件存储、对象存储、图床。
- 多服务器部署、微服务拆分、复杂消息队列、插件系统。

## 命名和状态收敛

- 模块名统一使用“每日内容配置”，不要写成“每日文案配置”。
- 飞书同步失败状态统一使用 `sync_failed`，不要新增 `feishu_failed`。
- 用户端展示来源是本地 `RECORD_DISPLAY` 和 `DAILY_RECORD` 状态，不直接读取飞书。
- 外部配置文件或环境变量保存敏感配置；数据库只保存非敏感配置和业务状态。
