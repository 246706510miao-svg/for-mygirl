# 运行与刷新说明

本文件说明本地 Docker 环境怎么启动、代码改动后怎么刷新，以及 MySQL/Redis 的 volume 怎么保留。

## 初次启动

从仓库根目录启动完整链路：

```powershell
docker compose --profile third-container --profile app up -d --build
```

完整 Docker 链路会启动一次性 `third-prompt-seed` 容器：它依赖 `third-migration` 成功完成，执行 `python -m third.scripts.seed_runagent_prompts`，把 `third/Prompt/runagent/*.yaml` 同步到 MySQL `prompt_registry` 后，`third-api` 和 `third-worker` 才会启动。本地直接运行 Python 时仍按 `third/README.md` 手动 seed。

Docker 前端默认不设置 `VITE_API_BASE_URL`，浏览器请求同源相对路径 `/api/...`，由本地 Caddy 反代到 `backend:8080`。只有前后端不同源部署时才设置完整后端源地址，不要设置成 `/api` 或默认 `http://localhost:8080`。

常用地址：

- 前端：`http://localhost:5173`（本地 Caddy，同源 `/api`）
- 后端：`http://localhost:8080`
- third API：`http://localhost:8001`
- MySQL：`127.0.0.1:3307`
- Redis：`127.0.0.1:6380`

## 改代码后是否每次都要重建

取决于你正在用哪种运行方式。

| 改动内容                                              | 本地直接运行                              | Docker 运行                                                                         |
| ----------------------------------------------------- | ----------------------------------------- | ----------------------------------------------------------------------------------- |
| `frontend/src`                                      | `npm run dev` 会热更新，通常不用重建    | 需要重建`frontend` 镜像，因为容器里跑的是已构建的 `dist`                        |
| `backend/src`                                       | 需要重启 SpringBoot 进程                  | 需要重建`backend` 镜像，因为 jar 在镜像构建时生成                                 |
| `backend/src/main/resources/db/migration`           | 重启后端，Flyway 启动时执行新 migration   | 重建并重启`backend`，Flyway 启动时执行新 migration                                |
| `third/` Python 代码                                | 本地`uvicorn --reload` 可热更新部分 API | 需要重建对应 third 容器镜像                                                         |
| `third/Prompt/runagent/*.yaml`                     | 执行 `python -m third.scripts.seed_runagent_prompts` | 完整 Docker 链路由 `third-prompt-seed` 自动同步；直接 Python 运行时仍需手动 seed |
| 文档                                                  | 不需要重启                                | 不需要重启                                                                          |
| 仅改运行环境变量                                      | 重启对应容器即可                          | 重启对应容器即可；但`VITE_API_BASE_URL` 是前端构建参数，改它需要重建 `frontend` |
| 仅改`third` 的 LLM key、代理、路由、探测或国内 provider 环境变量 | 重启本地`uvicorn` / worker 进程         | 重启`third-api` 和 `third-worker` 即可，不需要重建镜像                          |

## 常用刷新命令

只刷新前后端代码，保留 MySQL/Redis 数据：

```powershell
docker compose --profile third-container --profile app up -d --build backend frontend
```

如果想分两步看清构建和重启：

```powershell
docker compose --profile third-container --profile app build backend frontend
docker compose --profile third-container --profile app up -d backend frontend
```

只改后端或 migration：

```powershell
docker compose --profile third-container --profile app up -d --build backend
```

只改前端并且当前使用 Docker 前端：

```powershell
docker compose --profile third-container --profile app up -d --build frontend local-caddy
```

只重启容器，不重建镜像：

```powershell
docker compose --profile third-container --profile app restart backend frontend
```

注意：`restart` 不会把新代码打进镜像。接口新增后仍然报 `No static resource ...`，通常就是只重启了旧镜像，没有重新 `build`。

只改 `third/.env` 中的 LLM 出口环境变量时，重启 third 容器即可，例如：

- `OPENAI_API_KEY`
- `THIRD_OPENAI_PROXY_URL`
- `THIRD_OPENAI_TIMEOUT_SECONDS`
- `THIRD_OPENAI_MAX_RETRIES`
- `THIRD_LLM_ROUTE_MODE`
- `THIRD_LLM_FALLBACK_PROVIDERS`
- `THIRD_LLM_PROBE_*`
- `THIRD_DEEPSEEK_*`
- `THIRD_MINIMAX_*`

```powershell
docker compose --profile third-container --profile app restart third-api third-worker
```

验证 LLM 出口和国内兜底配置时，可以手动执行探测脚本；它只发送固定无业务内容：

```powershell
python -m third.scripts.probe_llm_routes --samples 3 --json --refresh
```

如果改了 `third/` Python 代码，例如 LLM route、OpenAI client 或 workflow 逻辑，需要重建 third 镜像：

```powershell
docker compose --profile third-container --profile app up -d --build third-api third-worker
```

## MySQL 和 Redis 是否需要刷新

通常不需要刷新数据服务。

- MySQL 表结构由 migration 管理：
  - `for_mygirl_app` 由 SpringBoot Flyway 管理，只允许 `backend_user` 访问。
  - `third_service` 由 `third-migration` 的 Alembic 管理，是 third 私有运行库，只允许 `third_user` 访问。
  - `third_test` 只给 third 的真实 MySQL 集成测试使用。
- `prompt_registry` 数据由 `third/Prompt/runagent/*.yaml` seed 得到。完整 Docker 链路用 `third-prompt-seed` 在启动链路中自动执行一次；本地直接运行 Python 时仍按 `third/README.md` 手动 seed。
- Redis 用于缓存、队列和运行时状态，通常跟随容器继续保留即可。
- third worker 会优先回收超过 `THIRD_WORKFLOW_PENDING_IDLE_MS` 的 Redis Stream pending 消息，超过 `THIRD_WORKFLOW_MAX_DELIVERIES` 后写入 `THIRD_WORKFLOW_DEAD_LETTER_QUEUE_NAME` 并把 workflow 标记失败。默认值已在代码里提供，生产不需要为了启用该能力单独上传 `third.env.prod`。
- `docker compose up -d --build backend frontend` 只重建应用镜像，不会删除 MySQL/Redis volume。

## 生产同步和卡住任务处理

生产代码同步走 GitHub Actions + GHCR。提交并 push 到 `main` 后，审批 `production` 部署，服务器会更新 `version.env`、拉取新镜像并 `up -d`。

4、5 这类修复上线时不要清 MySQL/Redis volume，也不要执行 `down -v`。保留 `prod_redis_data` 后，新 `third-worker` 会自动 reclaim 旧 pending 消息；只有需要调整 pending idle、最大投递次数或 dead-letter 队列名时，才手动上传更新后的 `deploy-private/prod/third.env.prod`。

## MySQL 权限边界修复

如果本地 MySQL volume 是旧版本创建的，可能还保留 `third_user` 访问业务库 `for_mygirl_app` 的授权。不要删除 volume，直接让 `mysql-init` 重新执行或手动执行同等 SQL：

```powershell
docker compose up -d mysql mysql-init
```

验证边界时应满足：

- `backend_user` 只能访问 `for_mygirl_app`。
- `third_user` 只能访问 `third_service` 和 `third_test`。
- SpringBoot 不能连接 `third_service`/`third_test`；third 不能连接 `for_mygirl_app`。

## volume 保留规则

Compose 当前使用两个命名 volume：

- `third_mysql_data`：保存 MySQL 数据。
- `third_redis_data`：保存 Redis 数据。

这些命令会保留 volume：

```powershell
docker compose up -d
docker compose up -d --build backend frontend
docker compose restart mysql redis
docker compose down
```

这个命令会删除 volume，开发时不要随手执行：

```powershell
docker compose down -v
```

只有明确想清空 MySQL/Redis 数据、重新跑所有初始化和 migration 时，才使用 `down -v`。

## 只清 Redis 不删 volume

如果只是 Redis 缓存或队列状态干扰调试，可以清 Redis 数据库，不需要删 volume：

```powershell
docker exec for-mygirl-third-redis-1 redis-cli FLUSHDB
```

这会清空 Redis 当前库的数据，但不会删除 `third_redis_data` volume。不要在还有重要队列任务时执行。

## 验证当前是否是新后端

看后端是否识别到当前 migration：

```powershell
docker logs --tail 120 for-mygirl-third-backend-1
```

日志里应该能看到类似：

```text
Successfully validated 8 migrations
Current version of schema `for_mygirl_app`: 8
```

验证积分接口时先走正式账号密码登录，让 PowerShell 保存后端写入的 Cookie，再复用同一个 web session 请求：

```powershell
Invoke-RestMethod -Method Post -Uri http://localhost:5173/api/auth/login -ContentType 'application/json' -Body '{"loginName":"legacy_user","password":"legacy-user-dev-password"}' -SessionVariable appSession
Invoke-WebRequest -Uri http://localhost:5173/api/points/summary -UseBasicParsing -WebSession $appSession
```

如果返回 `No static resource api/points/summary`，说明运行中的后端没有加载当前接口代码，优先重建并重启 `backend`。
