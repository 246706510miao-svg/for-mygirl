
## 1）最重要的阻塞问题

1. **后台 workflow 任务推进入口需要立刻确认。** 后端现在会把 third 调用写入 `RECORD_WORKFLOW_TASK`，并返回 `pollAfterMs=1500` 给前端轮询，但我在已读代码里只看到 `processPendingWorkflowTasks()` 和 `processWorkflowTask()`，没有确认到实际 `@Scheduled` 调度类调用它。若没有调度器，前端会一直看到 processing。
2. **root `docker-compose.yml` 仍没有 `third-prompt-seed`。** 文档说生产私有 Compose 有 seed，但当前根 compose 只有 `third-migration`、`third-api`、`third-worker`，没有 seed service；LLM 模式依赖 MySQL `prompt_registry`，缺 seed 会导致 Agent prompt 读取失败。
3. **Compose 前端构建参数仍会把 API 固定成 `http://localhost:8080`。** 前端代码默认相对 `/api` 已经修好，但 compose 又传了 `FRONTEND_API_BASE_URL:-http://localhost:8080`，部署到手机/域名时容易打到访问者自己的 localhost。

## 2）可能影响后续扩展的问题

* **third worker 没有 pending reclaim / dead-letter。** Redis Stream 只消费 `>` 新消息，worker 只有 `run_session` 正常返回后才 ack；异常时释放锁但不 ack，消息可能留在 pending。
* **后端仍保留 `invokeAndWait/resumeAndWait`。** 主记录链路已经不用它，但后续新增模块如果误用，会重新引入同步阻塞。
* **测试覆盖仍偏主干外侧。** 我确认到认证测试，但没有确认到记录 workflow 任务推进、third snapshot、确认门、Compose seed 的自动化测试。

## 3）建议优先处理顺序

1. 先确认并补上 `RECORD_WORKFLOW_TASK` 调度器。
2. root compose 增加 `third-prompt-seed`，并让 `third-api/third-worker` 依赖它完成。
3. compose 前端默认 API 改成空值或 `/api`，生产交给 Caddy 反代。
4. 给 third Redis worker 增加 pending reclaim、最大重试和 dead-letter。
5. 清理或标注 `invokeAndWait/resumeAndWait` 为 legacy，避免新模块误用。

## 4）需要你补充确认的信息

* 你现在本地是否有一个我没读到的调度类在定时调用 `recordService.processPendingWorkflowTasks()`。
* 生产部署是否确实会使用“私有 Compose”；如果只用当前根 compose，就必须补 seed。
* 生产前端是否计划同域名 `/api` 反代后端。
