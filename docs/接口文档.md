# 接口文档

本文档根据现有 `docs` 中的 UI 页面说明、页面流程图、活动图、架构图和数据库设计整理。接口优先覆盖 MVP 的完整记录闭环：用户输入、AI 草稿、用户确认、飞书同步、本地展示、后台人员异常处理和每日内容配置。

本文档主要给 Codex 和开发实现使用，不是面向最终用户的产品说明。读这个文档时，优先关注“一个记录如何被追踪”：前端请求带来哪些 ID，后端写了哪些表，状态如何变化，third workflow 和飞书同步失败时从哪里定位。

## Codex 使用口径

- 先按“记录追踪”确认一次记录的主键链路，再看具体接口。
- 实现 Controller 时，不只看请求和响应，还要同步实现幂等、状态流转、表写入和错误保留。
- 排查问题时，从 `requestId`、`sessionId`、`recordId`、`thirdSessionId`、`syncId` 反查，不要直接猜测是 AI、飞书还是本地保存失败。
- 用户端展示永远以本地 `RECORD_DISPLAY` 为准；飞书只是同步目标。
- `third` 返回的 workflow 结果不能原样给前端，必须由 SpringBoot 转成业务 DTO 并落本地业务表。

## 记录追踪

### 追踪目标

记录追踪用于回答四类问题：

| 问题 | 应查内容 |
|---|---|
| 用户说了什么 | `RECORD_SESSION`、`RECORD_MESSAGE`、`clientMessageId`。 |
| AI 生成了什么草稿 | `RECORD_DRAFT`、`versionNo`、`currentDraftId`。 |
| 用户最终确认了哪一版 | `clientConfirmId`、`final_draft_id`、`DAILY_RECORD`。 |
| 飞书为什么成功或失败 | `FEISHU_SYNC`、`payload_json`、`error_message`、`thirdSessionId`、third artifacts。 |

### 追踪 ID

| ID | 来源 | 保存位置 | 用途 |
|---|---|---|---|
| `requestId` | 前端请求头或后端生成 | 日志、响应体 | 串起一次 HTTP 调用。 |
| `sessionId` | `POST /api/record-sessions` 返回 | `RECORD_SESSION`、`RECORD_MESSAGE`、`RECORD_DRAFT` | 串起一次对话式记录过程。 |
| `clientMessageId` | 前端生成 | `RECORD_MESSAGE` 或幂等记录 | 防止重复消息和重复草稿。 |
| `draftId` | 后端生成 | `RECORD_DRAFT`、`RECORD_SESSION.current_draft_id`、`DAILY_RECORD.final_draft_id` | 标识某一版 AI 草稿。 |
| `clientConfirmId` | 前端生成 | `DAILY_RECORD` 或幂等记录 | 防止重复确认和重复写飞书。 |
| `recordId` | 确认写入后生成 | `DAILY_RECORD`、`RECORD_DISPLAY`、`FEISHU_SYNC` | 串起正式记录、展示数据和同步状态。 |
| `syncId` | 写入或重试飞书时生成 | `FEISHU_SYNC` | 标识一次飞书同步尝试。 |
| `thirdSessionId` | SpringBoot 调用 `third` 后返回 | 业务侧关联字段或日志、third session 表 | 追踪 workflow plan、step、artifact 和 Tool 结果。 |

实现时如果现有表没有显式字段保存 `clientMessageId`、`clientConfirmId`、`thirdSessionId`，需要补字段或补一张幂等/追踪表，不能只靠日志。

### 主链路追踪表

| 阶段 | 入口接口 | 必须记录 | 状态变化 | 失败时保留 |
|---|---|---|---|---|
| 创建会话 | `POST /api/record-sessions` | `sessionId`、`userId`、`recordDate` | `RECORD_SESSION.status=editing` | 创建失败原因和 `requestId`。 |
| 用户输入 | `POST /api/record-sessions/{sessionId}/messages` | `clientMessageId`、`RECORD_MESSAGE` | `editing -> previewing` 或继续 `editing` | 原始输入、AI 追问或错误原因。 |
| 语音输入 | `POST /api/record-sessions/{sessionId}/voice-messages` | `clientMessageId`、`asrText`、`RECORD_MESSAGE` | `editing -> previewing` 或继续 `editing` | ASR 错误、原始请求和 `requestId`。 |
| 生成草稿 | SpringBoot 内部调用 `third` | `thirdSessionId`、`draftId`、`versionNo` | 旧草稿 `replaced`，新草稿 `active` | third 状态、AI 错误、无效草稿 artifact。 |
| 用户确认 | `POST /api/record-sessions/{sessionId}/confirm` | `clientConfirmId`、`draftId` | `previewing -> confirming` | 幂等冲突、草稿不存在、状态冲突。 |
| 生成 payload | SpringBoot 内部调用 `third` | `thirdSessionId`、payload 候选 | `generating_payload -> validating` | 校验错误、AI 修正记录。 |
| 写入飞书 | third Tool 调飞书 | `syncId`、`payload_json`、`feishu_ref_id` | `syncing -> success` 或 `sync_failed` | payload 快照、飞书错误、重试次数。 |
| 展示落库 | 确认接口尾部 | `recordId`、`RECORD_DISPLAY` | `success/sync_failed/blocked -> displayed` | 展示状态和用户可见摘要。 |
| 后台人员重试 | `POST /api/admin/records/{recordId}/retry-sync` | 新 `syncId`、重试 mode | `sync_failed -> syncing -> success/sync_failed` | 每次重试的 payload、错误、次数。 |

### 状态检查点

| 检查点 | 正常条件 |
|---|---|
| `RECORD_SESSION.current_draft_id` | 指向最新 `active` 草稿。 |
| `RECORD_DRAFT.version_no` | 同一个 `sessionId` 下递增且唯一。 |
| `DAILY_RECORD.final_draft_id` | 指向用户最终确认的 `confirmed` 草稿。 |
| `DAILY_RECORD.status` | 只能是 `success`、`sync_failed`、`blocked`。 |
| `RECORD_DISPLAY.display_status` | 与 `DAILY_RECORD.status` 保持可展示映射。 |
| `FEISHU_SYNC.payload_json` | 飞书失败时也必须保留。 |
| `FEISHU_SYNC.retry_count` | 必须有上限，不能无限重试。 |

### 排查顺序

1. 有 `requestId` 时，先查 HTTP 日志，确认用户调的是哪个接口。
2. 有 `sessionId` 时，查 `RECORD_SESSION`，再查同 session 下的 `RECORD_MESSAGE` 和 `RECORD_DRAFT`。
3. 有 `recordId` 时，查 `DAILY_RECORD`，再查 `RECORD_DISPLAY` 和最新 `FEISHU_SYNC`。
4. 有 `thirdSessionId` 时，查 third 的 workflow session、step、artifact，确认卡在 plan、Agent、校验还是 Tool。
5. 如果用户端展示异常，最后再查飞书；用户端页面本身不直接依赖飞书。

## 1. 接口分层

### 1.1 前端业务 API

前端只调用 SpringBoot 业务 API，统一使用 `/api` 前缀。

| 调用方 | 被调用方 | 说明 |
|---|---|---|
| 用户端前端 | SpringBoot API | 首页、记录对话、最近记录。 |
| 后台运维前端 | SpringBoot API | 后台首页、每日内容配置、记录列表、记录详情、重试同步、展示修正。 |

### 1.2 后端内部 third API

SpringBoot 通过 HTTP 调用 Python `third` 服务。前端不直接调用 `third`。

| 调用方 | 被调用方 | 说明 |
|---|---|---|
| SpringBoot API | `third` Workflow API | 提交 workflow、查询状态、resume 确认门。 |
| `third` worker | OpenAI API | workflowagent 和业务 Agent 调用。 |
| `third` worker | 飞书 OpenAPI | 通过 Tool 写入飞书多维表格或文档。 |

### 1.3 页面到接口

| 页面 | 主要接口 |
|---|---|
| 用户首页 | `GET /api/user/home` |
| 记录对话页 | `POST /api/record-sessions`、`POST /api/record-sessions/{sessionId}/messages`、`POST /api/record-sessions/{sessionId}/voice-messages`、`GET /api/record-sessions/{sessionId}`、`POST /api/record-sessions/{sessionId}/confirm`、`POST /api/record-sessions/{sessionId}/cancel` |
| 最近记录页 | `GET /api/user/records` |
| 后台首页 | `GET /api/admin/dashboard` |
| 每日内容配置页 | `GET /api/admin/daily-contents`、`PUT /api/admin/daily-contents` |
| 用户最近记录页 | `GET /api/admin/records` |
| 记录详情页 | `GET /api/admin/records/{recordId}`、`POST /api/admin/records/{recordId}/retry-sync`、`PATCH /api/admin/records/{recordId}/display` |

## 2. 基础约定

### 2.1 服务地址

| 环境 | Base URL | 说明 |
|---|---|---|
| 本地开发 | `http://localhost:8080/api` | SpringBoot 业务 API。 |
| 生产环境 | `https://<domain>/api` | Nginx 转发到业务后端。 |
| third 本地开发 | `http://127.0.0.1:8001` | SpringBoot 内部调用或本地调试使用。 |

### 2.2 数据格式

- 请求体默认使用 `application/json`。
- 语音上传接口使用 `multipart/form-data`。
- 响应体使用 `application/json`。
- 日期使用 `yyyy-MM-dd`，例如 `2026-06-04`。
- 时间使用 ISO 8601，例如 `2026-06-04T21:30:00+08:00`。
- ID 使用字符串，由后端生成，建议 UUID、雪花 ID 或带前缀业务 ID。

### 2.3 鉴权

MVP 使用两个固定普通用户和一个固定后台人员账号登录。登录成功后，前端在后续请求中携带：

```http
Authorization: Bearer <accessToken>
```

| 角色 | 允许访问 |
|---|---|
| `USER` | 手机端用户视角、绑定管理员视角、记录会话、最近记录、评论打分、积分奖品和兑换记录。开发账号为 `user`、`partner`。 |
| `OPS_ADMIN` | 后台首页、每日内容配置、用户记录查看、同步重试、展示修正。当前接口路径仍兼容 `/api/admin/*`。 |

手机端“管理员视角”不是 `OPS_ADMIN`，而是普通 `USER` 的 `currentViewRole=BOUND_ADMIN`。它只能操作绑定对象的数据，不能访问 `/api/admin/*`。

### 2.4 通用请求头

| Header | 必填 | 说明 |
|---|---:|---|
| `Authorization` | 是 | 登录后的 Bearer token。 |
| `X-Request-Id` | 否 | 调用链追踪 ID；不传时后端生成。 |

### 2.5 统一响应

成功响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {},
  "requestId": "req_20260604_001"
}
```

失败响应：

```json
{
  "code": "VALIDATION_ERROR",
  "message": "请求参数不合法",
  "details": [
    {
      "field": "content",
      "reason": "不能为空"
    }
  ],
  "requestId": "req_20260604_001"
}
```

### 2.6 常用错误码

| code | HTTP 状态码 | 说明 |
|---|---:|---|
| `OK` | 200 | 成功。 |
| `CREATED` | 201 | 创建成功。 |
| `BAD_REQUEST` | 400 | 请求格式错误。 |
| `UNAUTHORIZED` | 401 | 未登录或 token 失效。 |
| `FORBIDDEN` | 403 | 当前角色无权限。 |
| `NOT_FOUND` | 404 | 资源不存在。 |
| `CONFLICT` | 409 | 状态冲突、重复提交或幂等冲突。 |
| `VALIDATION_ERROR` | 422 | 字段校验失败。 |
| `AI_SERVICE_ERROR` | 502 | AI 服务异常。 |
| `ASR_SERVICE_ERROR` | 502 | 语音识别服务异常。 |
| `FEISHU_SYNC_ERROR` | 502 | 飞书同步异常。 |
| `INTERNAL_ERROR` | 500 | 服务内部错误。 |

### 2.7 幂等约定

| 字段 | 使用接口 | 作用 |
|---|---|---|
| `clientMessageId` | 文本消息、语音消息 | 防止重复写入 `RECORD_MESSAGE` 和重复生成草稿。 |
| `clientConfirmId` | 确认写入 | 防止重复创建 `DAILY_RECORD` 和重复写入飞书。 |

实现要求：

- `clientMessageId` 需要和 `sessionId` 建唯一约束或等价幂等记录。
- `clientConfirmId` 需要和 `sessionId` 建唯一约束或等价幂等记录。
- 重复提交同一个幂等键时，返回第一次处理结果。

### 2.8 分页结构

```json
{
  "items": [],
  "page": 1,
  "pageSize": 20,
  "total": 36
}
```

## 3. 通用数据结构

### 3.1 Person

```json
{
  "id": "person_user",
  "role": "USER",
  "displayName": "fjl",
  "enabled": true
}
```

### 3.2 RecordSession

```json
{
  "id": "session_001",
  "status": "previewing",
  "currentDraftId": "draft_002",
  "createdAt": "2026-06-04T21:00:00+08:00",
  "updatedAt": "2026-06-04T21:05:00+08:00"
}
```

`status` 可选值：

| 值 | 说明 |
|---|---|
| `editing` | 用户正在输入或继续修改。 |
| `previewing` | 已生成草稿，等待继续修改或确认。 |
| `confirming` | 用户已确认，后端正在生成 payload 或同步飞书。 |
| `confirmed` | 确认流程结束，已形成正式记录。 |
| `cancelled` | 用户取消本次记录。 |

```json
        "type": "feishu.sync.failed",
        "refId": "sync_001",
        "summary": "飞书同步失败，已保留 payload"
      }
    ],
    "issues": []
  },
  "requestId": "req_trace_001"
}
```

主要读表：

- `RECORD_SESSION`
- `RECORD_MESSAGE`
- `RECORD_DRAFT`
- `DAILY_RECORD`
- `RECORD_DISPLAY`
- `FEISHU_SYNC`
- third workflow session、step、artifact 表或其聚合结果

规则：

- 这个接口只读，不修复数据。
- `payload_json` 默认不要完整返回给前端页面；调试模式需要查看时再显式打开或做脱敏。
- `issues` 用于返回追踪发现的问题，例如 currentDraft 丢失、finalDraft 不存在、display 缺失、sync 没有 payload。
- 如果没有保存 `thirdSessionId` 关联，只能返回业务表追踪，并在 `issues` 中提示缺少 workflow 关联。

### 8.2 查询会话追踪

```http
GET /api/admin/record-sessions/{sessionId}/trace
```

等价于：

```http
GET /api/admin/record-traces?sessionId={sessionId}
```

用于排查用户还没有确认、只停留在草稿阶段的问题。

### 8.3 查询正式记录追踪

```http
GET /api/admin/records/{recordId}/trace
```

等价于：

```http
GET /api/admin/record-traces?recordId={recordId}
```

用于排查已经确认后的记录、展示和飞书同步问题。

### 8.4 追踪接口实现提示

- 聚合结果可以由 SpringBoot 查询业务库，再调用 third 查询 workflow 状态和 artifact 摘要。
- timeline 不一定需要单独建事件表，MVP 可以从各表的 `created_at`、`updated_at`、`confirmed_at`、`last_sync_at` 推导。
- 如果后续需要更精确的审计，再补 `RECORD_TRACE_EVENT` 表保存事件。
- 追踪接口不能改变业务状态；重试和修正仍然走 `retry-sync`、`display` 等明确的写接口。

## 9. SpringBoot 到 third 的内部接口

以下接口不是前端业务接口，只给 SpringBoot 适配层调用。`third` 的本地默认地址是 `http://127.0.0.1:8001`。

### 9.1 提交 workflow

```http
POST /workflows/invoke
```

请求：

```json
{
  "content": [
    {
      "text": "根据当前记录会话生成草稿或写入飞书。"
    }
  ],
  "metadata": {
    "businessSessionId": "session_001",
    "operation": "draft_generate",
    "idempotencyKey": "cmid_20260604_0001"
  }
}
```

响应：

```json
{
  "session_id": "sess_xxx",
  "status": "queued",
  "confirmation": null,
  "content": [
    {
      "text": ""
    }
  ],
  "error_text": null
}
```

### 9.2 查询 workflow 状态

```http
GET /workflows/{session_id}
```

响应字段沿用 `third` 服务返回：

```json
{
  "session_id": "sess_xxx",
  "status": "success",
  "confirmation": null,
  "content": [
    {
      "text": "workflow result"
    }
  ],
  "error_text": null
}
```

### 9.3 恢复确认门

```http
POST /workflows/{session_id}/resume
```

请求：

```json
{
  "confirmation_id": "confirm_xxx",
  "approved": true,
  "content": [
    {
      "text": "用户已经在业务前端确认写入。"
    }
  ]
}
```

### 9.4 适配规则

- SpringBoot 负责把业务 session、draft、飞书目标、字段 Schema、payload 规范整理成 `third` 输入。
- `third` 负责 workflow plan、Agent 调用、校验节点、Tool 分发和飞书调用。
- SpringBoot 不能把 `third` 的 artifact 原样透传给前端，必须转换成业务 DTO。
- 写入类 workflow 如果进入 `waiting_user`，SpringBoot 可以根据用户已经完成的业务确认调用 `/resume`。
- `third` 失败时，SpringBoot 必须把错误映射到 `sync_failed` 或 `blocked`，并保存错误原因。

## 12. 01-优化 MVP 手机端绑定接口

本节以 `docs/future/01-优化mvp功能.md` 为准，补充当前手机端双向绑定 MVP。旧后台运维接口继续属于 `ops`，不进入绑定管理员视角。

### 12.1 登录与视角

| 接口 | 模块 | 说明 |
|---|---|---|
| `POST /api/auth/login` | `identity` | 支持 `user`、`partner`、`admin` 三个开发账号。 |
| `GET /api/auth/me` | `identity` + `relationship` | 返回当前登录人、当前视角、视角数据拥有者和绑定关系。 |
| `POST /api/identity/view-role` | `identity` + `relationship` | 普通用户在 `USER` 和 `BOUND_ADMIN` 之间切换。 |
| `GET /api/relationship/binding` | `relationship` | 返回绑定对象和授权能力。 |

`GET /api/auth/me` 响应核心结构：

```json
{
  "person": { "id": "person_user", "role": "USER", "displayName": "fjl", "enabled": true },
  "currentViewRole": "USER",
  "viewOwner": { "id": "person_user", "role": "USER", "displayName": "fjl", "enabled": true },
  "binding": {
    "active": true,
    "boundUser": { "id": "person_partner", "role": "USER", "displayName": "TA", "enabled": true },
    "permissions": ["record.comment", "reward.manage"]
  }
}
```

### 12.2 最近记录与评论打分

| 接口 | 模块 | 说明 |
|---|---|---|
| `GET /api/records/recent` | `record` + `comment` | 当前用户查看自己的最近记录，返回绑定评论和绑定用户打分。 |
| `GET /api/bound-user/records/recent` | `relationship` + `record` + `comment` | 绑定管理员视角查看绑定对象最近记录。 |
| `POST /api/records/{recordId}/comment` | `comment` + `points` | 绑定管理员视角保存评论和 0-100 分打分；评论打分只落本地库，不写飞书。 |

打分积分规则：

- 保存打分时读取 `DAILY_RECORD.ai_score` 或展示分数。
- 记录积分为 `floor((AI 分数 * 50% + 绑定用户打分 * 50%) / 10)`，最高 10 分。
- 每条记录只有一条 `record_score` 积分来源；再次改分只按差额调整。
- 每日积分上限自然由 `当天记录事项数 * 10 + 签到 10` 构成。

### 12.3 积分、奖品和兑换记录

| 接口 | 模块 | 说明 |
|---|---|---|
| `POST /api/points/checkins` | `points` | 当前登录用户每日签到，幂等增加 10 积分。 |
| `GET /api/points/summary` | `points` | 返回当前登录用户积分和当前视角数据拥有者积分。 |
| `GET /api/rewards` | `points` | 返回当前视角可见的未兑换奖品。 |
| `POST /api/rewards` | `relationship` + `points` | 绑定管理员视角给绑定对象添加奖品。 |
| `POST /api/rewards/{rewardId}/redeem` | `points` | 用户兑换自己的奖品，扣积分并把奖品标记为已兑换。 |
| `GET /api/reward-redemptions` | `points` | 返回当前视角数据拥有者的兑换记录；绑定管理员视角用于显示兑换提示。 |

奖品兑换规则：

- 默认奖品通过 migration 内置给两个普通用户。
- `REWARD_ITEM.status=active` 的奖品才会展示在可兑换列表。
- 兑换成功后写 `POINT_LEDGER` 负数流水，写 `REWARD_REDEMPTION`，并把奖品状态改为 `redeemed`。
- 评论、打分、签到、积分、奖品和兑换记录都不进入 `FEISHU_SYNC`。
