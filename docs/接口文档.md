# 接口文档

本文档根据现有 UI 原型、活动图、架构图和数据库设计整理。接口优先覆盖 MVP 的完整记录闭环：用户输入、AI 草稿、用户确认、飞书同步、本地展示、管理员处理异常和配置每日内容。

## 1. 基础约定

### 1.1 服务地址

| 环境     | Base URL                  | 说明                 |
| -------- | ------------------------- | -------------------- |
| 本地开发 | `http://localhost:8080` | Spring Boot 开发环境 |
| 生产环境 | `https://<domain>`      | Nginx 转发到后端服务 |

所有业务接口统一使用 `/api` 前缀。

### 1.2 数据格式	

- 请求体：`application/json`
- 响应体：`application/json`
- 时间格式：ISO 8601，例如 `2026-06-04T21:30:00+08:00`
- 日期格式：`yyyy-MM-dd`，例如 `2026-06-04`
- ID 类型：字符串，由后端生成，建议使用 UUID 或雪花 ID

### 1.3 鉴权

MVP 可以先用固定账号登录，后续再扩展短信、密码或第三方登录。

前端登录成功后，在后续请求中携带：

```http
Authorization: Bearer <accessToken>
```

权限边界：

| 角色      | 允许访问                                                       |
| --------- | -------------------------------------------------------------- |
| `USER`  | 用户首页、记录会话、最近记录                                   |
| `ADMIN` | 管理员首页、每日内容配置、用户记录查看、同步重试、展示内容修正 |

### 1.4 统一响应结构

成功响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {}
}
```

失败响应：

```json
{
  "code": "VALIDATION_ERROR",
  "message": "请求参数不合法",
  "details": [
    {
      "field": "content",
      "reason": "不能为空"
    }
  ]
}
```

### 1.5 常用错误码

| code                  | HTTP 状态码 | 说明                |
| --------------------- | ----------: | ------------------- |
| `OK`                |         200 | 成功                |
| `CREATED`           |         201 | 创建成功            |
| `BAD_REQUEST`       |         400 | 请求格式错误        |
| `UNAUTHORIZED`      |         401 | 未登录或 token 失效 |
| `FORBIDDEN`         |         403 | 当前角色无权限      |
| `NOT_FOUND`         |         404 | 资源不存在          |
| `CONFLICT`          |         409 | 状态冲突或幂等冲突  |
| `VALIDATION_ERROR`  |         422 | 字段校验失败        |
| `AI_SERVICE_ERROR`  |         502 | 大模型服务异常      |
| `ASR_SERVICE_ERROR` |         502 | 语音识别服务异常    |
| `FEISHU_SYNC_ERROR` |         502 | 飞书同步异常        |
| `INTERNAL_ERROR`    |         500 | 服务内部错误        |

### 1.6 幂等约定

以下客户端 ID 必须持久化，用于防止前端重试造成重复数据：

| 字段                | 使用接口                   | 作用                                           |
| ------------------- | -------------------------- | ---------------------------------------------- |
| `clientMessageId` | 发送文本消息、发送语音消息 | 防止重复写入 `RECORD_MESSAGE` 和重复生成草稿 |
| `clientConfirmId` | 确认写入                   | 防止重复创建 `DAILY_RECORD` 和重复写入飞书   |

建议后端建立唯一约束：

- `record_message(session_id, client_message_id)`
- `daily_record(session_id, client_confirm_id)` 或等价幂等表

## 2. 通用数据结构

### 2.1 Person

```json
{
  "id": "person_001",
  "role": "USER",
  "displayName": "fjl",
  "enabled": true
}
```

### 2.2 RecordSession

```json
{
  "id": "session_001",
  "status": "previewing",
  "currentDraftId": "draft_002",
  "createdAt": "2026-06-04T21:00:00+08:00",
  "updatedAt": "2026-06-04T21:05:00+08:00"

```

`status` 可选值：

| 值             | 说明                                |
| -------------- | ----------------------------------- |
| `editing`    | 用户正在输入或继续修改              |
| `previewing` | 已生成草稿，等待继续修改或确认      |
| `confirming` | 用户已确认，正在生成 payload 或同步 |
| `confirmed`  | 确认流程结束，已形成正式记录        |
| `cancelled`  | 用户取消                            |

## 5. 记录会话接口

### 5.1 创建记录会话

```http
POST /api/record-sessions
```

请求：

```json
{
  "recordDate": "2026-06-04",
  "source": "user_home"
}
```

响应：

```json
{
  "code": "CREATED",
  "message": "success",
  "data": {
    "id": "session_001",
    "status": "editing",
    "currentDraftId": null,
    "createdAt": "2026-06-04T21:00:00+08:00",
    "updatedAt": "2026-06-04T21:00:00+08:00"
  }
}
```

主要写表：

- `RECORD_SESSION`

业务规则：

- 创建后状态为 `editing`。
- 可以允许同一天多个 session，但一个 session 最终最多确认成一条 `DAILY_RECORD`。

- `RECORD_MESSAGE`
- `RECORD_DRAFT`

业务规则：

- 新增、修改、删除、补充都走同一个接口，由 AI 结合上下文判断意图。

- 每次 AI 生成或修改草稿都必须新增一条 `RECORD_DRAFT`，旧草稿状态改为 `replaced`。
- 如果 AI 无法理解，应返回 AI 追问消息，可以不生成新草稿。
- 重复提交相同 `clientMessageId` 时，直接返回第一次处理结果。

同步失败响应仍然是业务成功，因为本地记录已经保存：

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "record": {
      "id": "record_001",
      "status": "sync_failed"
    },
    "feishuSync": {
      "id": "sync_001",
      "syncStatus": "failed",
      "errorMessage": "飞书接口超时",
      "retryCount": 1
    },
    "display": {
      "recordId": "record_001",
      "displayStatus": "sync_failed",
      "summary": "记录已保存，飞书同步暂时失败。"
    }
  }
}
```

主要读写表：

- `RECORD_SESSION`
- `RECORD_DRAFT`
- `DAILY_RECORD`
- `FEISHU_SYNC`
- `RECORD_DISPLAY`
- `RECORD_MESSAGE`

业务规则：

- 必须确认当前 session 下存在且未确认过的 `draftId`。
- 确认时锁定草稿版本，`RECORD_DRAFT.status` 改为 `confirmed`。
- 后端准备 AI 上下文并生成飞书 MCP payload。
- AI 生成的 payload 必须经过后端校验，校验内容包括目标位置、字段 schema、字段类型、必要字段、权限和内容安全。
- 校验通过后保存 payload 快照，再调用飞书 MCP。
- 飞书失败不能丢本地记录，必须保存正式记录、payload、错误原因和展示状态。
- 重复提交同一个 `clientConfirmId` 时，返回第一次确认结果。

状态映射：

| 场景                   | `DAILY_RECORD.status` | `FEISHU_SYNC.sync_status` | `RECORD_DISPLAY.display_status` |
| ---------------------- | ----------------------- | --------------------------- | --------------------------------- |
| 本地保存和飞书同步成功 | `success`             | `success`                 | `success`                       |
| 本地保存成功，飞书失败 | `sync_failed`         | `failed`                  | `sync_failed`                   |
| payload 安全或权限异常 | `blocked`             | `failed` 或不创建         | `blocked`                       |

### 5.6 取消记录会话

```http
POST /api/record-sessions/{sessionId}/cancel
```

请求：

```json
{
  "reason": "user_cancelled"
}
```

响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "id": "session_001",
    "status": "cancelled",
    "updatedAt": "2026-06-04T21:06:00+08:00"
  }
}
```

主要写表：

- `RECORD_SESSION`

业务规则：

- 只有 `editing`、`previewing` 状态允许取消。
- 已进入确认或已确认的 session 不允许取消，应返回 `CONFLICT`。

### 6.6 重试飞书同步

```http
POST /api/admin/records/{recordId}/retry-sync
```

请求：

```json
{
  "mode": "reuse_payload"
}
```

`mode` 可选值：

| 值                     | 说明                              |
| ---------------------- | --------------------------------- |
| `reuse_payload`      | 使用上一次保存的 payload 直接重试 |
| `regenerate_payload` | 重新生成并校验 payload 后再写入   |

响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "recordId": "record_001",
    "recordStatus": "success",
    "feishuSync": {
      "id": "sync_002",
      "syncStatus": "success",
      "feishuRefId": "rec_xxx",
      "retryCount": 2,
      "lastSyncAt": "2026-06-04T22:00:00+08:00",
      "errorMessage": null
    },
    "displayStatus": "success"
  }
}
```

主要读写表：

- `DAILY_RECORD`
- `FEISHU_SYNC`
- `RECORD_DISPLAY`
- `APP_CONFIG`

业务规则：

- 只允许 `sync_failed` 或可修复的 `blocked` 记录重试。
- 重试次数需要有限制，不能无限重试。
- 每次重试建议新增一条 `FEISHU_SYNC`，保留历史失败原因。

## 7. 后端内部服务边界

以下不是前端直接调用的公网接口，但实现时需要清晰分层：

| 内部能力        | 调用方             | 说明                                             |
| --------------- | ------------------ | ------------------------------------------------ |
| AI 草稿生成     | 记录业务模块       | 根据 session 消息和当前草稿生成新草稿            |
| AI payload 生成 | 确认写入、重试同步 | 生成符合飞书 MCP 规范的 payload                  |
| Payload 校验    | 确认写入、重试同步 | 校验目标、schema、类型、必填字段、权限和内容安全 |
| ASR 语音识别    | 语音消息接口       | 把语音转成 `asrText`                           |
| 飞书 MCP 写入   | 确认写入、重试同步 | 写入飞书文档或多维表格                           |

外部服务失败时，后端必须记录错误原因，避免只向前端返回模糊失败。
